# Brain Fast Iteration Plan

## 目标

本文档给出一个面向 15 天极限开发周期的 `brain` 侧落地规划。目标不是做最先进的算法，而是在最短时间内，把以下能力做成“可跑、可测、可联调”的版本：

- 守门员队长机制
- 3v3 常规传球
- 二过一传切
- 守门员动态防守
- `brain` 执行决策与 `locator` 解耦

总体原则只有一条：

- 放弃复杂数学模型和机器学习，全部使用最直接的几何逻辑、状态机、硬阈值与超时保护。

## 为什么先拆模块

从当前仓库结构看，`brain` 还是一个单可执行体：

- [src/brain/CMakeLists.txt](../src/brain/CMakeLists.txt)

现在的问题不是功能完全没有，而是以下逻辑耦合得太紧：

- 决策逻辑主要散在 `Brain` 和 `brain_tree.cpp`
- 定位能力由 `Locator` 提供，但被行为树节点直接深度调用
- 多机协作逻辑混在 `Brain::handleCooperation()` 和 `brain_communication.cpp`
- 守门员、前锋、传球、找球、定位修正都挤在同一个运行体里

在 15 天冲刺下，这种结构最大的问题不是“不优雅”，而是：

- 改一个角色逻辑，容易碰到定位流程
- 改协作逻辑，容易碰到行为树黑板
- 场上联调时，故障边界不清晰
- 构建时间和回归范围太大

所以建议先把 `brain` 逻辑按“推理职责”拆成两个模块 build：

1. `brain_locator`
2. `brain_decision`

最终仍然可以由 `brain_node` 作为总装配入口，但内部链接两个独立模块。

## 推荐模块边界

### 模块一：`brain_locator`

职责只保留以下几件事：

- 标记点定位
- 场线定位辅助
- odom 与 field 坐标系转换
- 定位结果质量评估
- 定位参数与阈值管理

建议纳入：

- `locator.h / locator.cpp`
- `Brain` 中与 `odomToField`、`robotPoseToField` 更新直接相关的辅助逻辑
- 与定位校验相关的公共函数

不应继续放在该模块中的内容：

- 守门员判罚
- 前锋角色切换
- 传球与二过一状态机
- UDP 队友协作决策

### 模块二：`brain_decision`

职责只保留以下几件事：

- 行为树节点
- 黑板状态更新
- 队友角色分配
- 常规传球/二过一状态机
- 守门员动态防守
- 战术级超时与回退

建议纳入：

- `brain_tree.h / brain_tree.cpp`
- `brain_communication.*` 中和“协作判定”直接相关的部分
- `Brain::handleCooperation()`
- `Brain::updateCostToKick()`
- 新增的队长仲裁与传切状态机

### 总装配层：`brain_node`

顶层 `brain_node` 只做：

- ROS 订阅/发布
- `BrainData` 聚合
- 调用 `brain_locator`
- 调用 `brain_decision`
- 调用 `RobotClient`

目标是让顶层变成胶水层，而不是继续承担所有策略。

## 15 天开发顺序

如果时间极其紧，开发顺序必须按“先稳定基本面，再叠高风险配合”来排。

### 第 1 阶段：第 1 到第 3 天

目标：先把结构和数据通路铺出来，不碰复杂战术。

优先完成：

1. `brain` 模块边界确认
2. `brain_locator` / `brain_decision` 的 build 拆分
3. 队友状态广播数据包定稿
4. 守门员队长仲裁的最小闭环

这几天不要做：

- 二过一
- 守门员扑救
- 复杂传球筛选

只要先做到：

- 每台前场机器人稳定广播基础状态
- 守门员稳定收到状态
- 守门员能稳定广播 `Striker / Supporter` 指派
- 两个前场机器人收到后能强制切树

就已经成功了一半。

### 第 2 阶段：第 4 到第 7 天

目标：把最基础、最稳的协作逻辑做出来。

优先完成：

1. 守门员队长机制正式版
2. Supporter 拉开跑位
3. 常规传球触发判定
4. 常规传球后的身份平滑切换

这阶段的核心不是“传得漂亮”，而是：

- 不要角色抖动
- 不要传出去没人接
- 不要两个人一起扑球

具体优先级：

1. 先做角色稳定性
2. 再做 Supporter 跑位
3. 再做短传
4. 最后做接球切换

### 第 3 阶段：第 8 到第 10 天

目标：补守门员动态防守。

优先完成：

1. 守门员横向 Y 跟随
2. 守门员左右限幅
3. 球速阈值触发扑倒
4. 扑倒后延迟起身
5. 起身后重定位扫描

这一阶段不要追求“像真实门将一样聪明”，只要实现：

- 球在哪边，我就侧移到哪边
- 球危险地滚向门，我就按规则扑
- 扑完不要立刻起身

就已经是非常有价值的提升。

### 第 4 阶段：第 11 到第 13 天

目标：再上二过一传切。

原因是二过一时序最紧、风险最高，必须放在后面，建立在：

- 队长机制稳定
- 普通传球稳定
- Supporter 跑位稳定

之后再做。

优先完成：

1. `Trigger_Wall_Pass` 握手信号
2. Striker 传球后前插 1 米
3. Supporter 一脚回做
4. 3 秒超时退出
5. 速度椭圆裁切

### 第 5 阶段：第 14 到第 15 天

目标：不再扩功能，只做实战联调和阈值清洗。

重点做：

- 网络丢包测试
- 摔倒状态同步测试
- 角色切换迟滞测试
- 传球力道测试
- 守门员扑倒阈值测试
- 超时退出测试

最后两天必须默认“只修 bug，不上新功能”。

## 每个功能先写什么

下面是按代码落地顺序，而不是按战术重要性排列。

### 1. 先写队友基础状态结构

先做的数据字段：

- `robot_id`
- `pose_x`
- `pose_y`
- `ball_seen`
- `ball_distance`
- `robot_state`

可选但推荐预留：

- `ball_field_x`
- `ball_field_y`
- `role`
- `timestamp_ms`

原因：

- 后续守门员仲裁、常规传球、二过一、守门员协防，全都依赖这个基础结构。

### 2. 再写角色仲裁器

建议新增一个很薄的仲裁器模块，例如：

- `role_manager`
- 或 `captain_referee`

它只做：

- 比较前场两人到球距离
- 应用夺权门限
- 应用摔倒/丢球降级
- 输出 `striker_id` / `supporter_id`

不要把行为树执行放进这里。

### 3. 再写 Supporter 跑位几何逻辑

建议单独抽一个几何目标生成器，例如：

- `support_position_planner`

输入：

- Striker 当前 `x/y`
- 球当前 `x/y`
- 场地宽度

输出：

- Supporter 目标点

规则只用硬阈值：

- 与 Striker 横向拉开 `1.5m ~ 2.0m`
- 纵向平齐或微突前 `0.2m`
- 不要越界

### 4. 再写常规传球状态机

建议先别塞进大行为树深处，先单独做一个有限状态机：

- `IDLE`
- `EVALUATE_PASS`
- `PASSING`
- `WAIT_RECEIVE_SWITCH`
- `EXIT`

这样后面调试日志更清楚。

### 5. 最后才写二过一状态机

状态建议更严格：

- `IDLE`
- `ARMED`
- `PASS_AND_GO`
- `ONE_TOUCH_RETURN`
- `WAIT_REACQUIRE`
- `TIMEOUT_EXIT`

一定要把超时退出写在状态机骨架里，而不是最后补。

## 守门员队长机制的实现建议

### 目标

守门员只做裁判和防守，不参与进攻推理。

### 最小决策输入

前场每个队员只看：

- 是否活着
- 是否摔倒
- 是否看得到球
- 到球距离

### 最小决策输出

- `striker_id`
- `supporter_id`
- 是否触发强制降级

### 必须先做的保护

`夺权门限`

- 建议先固定成 `0.30m`
- 只有新候选者比当前 Striker 近至少 `0.30m` 才换人

`丢球降级`

- 连续 `1.0s` 丢球，直接丧失 Striker 资格

`摔倒降级`

- 一旦状态为 `fallen`，直接降到最低优先级

`无条件服从`

- 收到守门员角色广播后，前场机器人必须强制切行为树分支

不要在前场本地再“民主表决”。

## 常规传球的实现建议

### 触发优先级

建议优先级低于：

- 近距离直接射门
- 明显空门推进

建议优先级高于：

- 被阻挡还继续强行带球

### 传球触发条件

全部用硬条件：

- Striker 前方有障碍
- Supporter 已就位
- 通信连续稳定若干帧
- Supporter 未摔倒
- 球在可控脚下范围

### 传球目标

不要传脚下，必须传前置量：

- Supporter 当前点前方 `0.4m ~ 0.5m`

### 传球后角色切换

建议顺序：

1. Striker 本地清除持球主攻身份
2. Striker 广播“已传球”
3. Supporter 感知到来球并接管
4. 守门员仲裁确认新角色

这样即使通信有轻微延迟，也不至于两个人都卡住。

## 二过一的实现建议

### 为什么必须最后做

因为它同时要求：

- 通信时序紧
- 步态速度高
- 接球不停车
- 回传方向硬编码准确

这是 15 天内最容易拖垮进度的部分。

### 最小触发条件

- Striker 与防守者距离 `< 0.6m`
- Supporter 与 Striker 距离 `< 1.5m`
- Supporter 在可接应区域
- 通信正常
- 两者都未摔倒

### 必做保护

`3 秒超时退出`

- 一旦触发二过一，3 秒内没人重新控球，就全员回基础追球状态

`步态椭圆限幅`

- 必须先做，不然前插摔倒概率太高

`回做动作硬编码`

- 不要追求停球后再传
- 直接用单次回做动作

## 守门员动态防守的实现建议

### 第一版只做两个动作

1. 横向侧移对齐球
2. 危险条件下按左右/中间扑倒

### X/Y 控制原则

`X` 轴：

- 固定在门线前 `0.4m`

`Y` 轴：

- 跟随球 `y`
- 但要限制在门柱内缩 `0.2m`

### 扑倒触发条件

建议全用硬阈值：

- 球进入己方禁区线以内
- 球朝己方球门方向滚动
- 球 `x` 方向速度超过阈值

### 扑倒方向

- 球在身体左侧 `> 0.15m`：左扑
- 球在身体右侧 `> 0.15m`：右扑
- 否则：中间扑或跪地扩面

### 扑倒后流程

1. 动作执行
2. 地上锁死 `2.5s ~ 3s`
3. 再起身
4. 起身后 `1s` 摆头扫描重定位

这个顺序不要颠倒。

## 推荐的代码改造顺序

### 第一步：先拆 build，不先改算法

建议把 `src/brain/CMakeLists.txt` 从“单一可执行文件吃掉所有 `src/*.cpp`”改成：

- `add_library(brain_locator ...)`
- `add_library(brain_decision ...)`
- `add_executable(brain_node ...)`

然后：

- `brain_node` 链接 `brain_locator`
- `brain_node` 链接 `brain_decision`

### 第二步：把 `Locator` 从行为树直接依赖中抽薄

当前 `brain_tree.cpp` 里多处直接调用：

- `brain->locator->locateRobot(...)`
- `brain->locator->residual(...)`

建议先加一层接口，例如：

- `LocalizationService`
- 或 `BrainLocalizationFacade`

让行为树只依赖“定位结果”和“定位请求接口”，而不是直接操作 `Locator` 细节。

### 第三步：把协作判定从 `Brain::handleCooperation()` 抽成单独模块

新增建议模块：

- `team_role_manager`
- `team_tactics_fsm`

不要继续把协作状态和底层 ROS 回调都堆在 `Brain` 主类里。

## 你现在应该先做什么

如果只允许从明天开始立刻动工，我建议按下面顺序：

1. 先拆 `brain_locator` / `brain_decision` 两个 build 模块。
2. 再定义统一队友状态数据结构和广播频率。
3. 再落守门员队长仲裁器。
4. 再落 Supporter 跑位几何逻辑。
5. 再落常规传球状态机。
6. 再落守门员横向动态防守。
7. 最后才做二过一。

原因很直接：

- 队长机制是所有多人协作的入口
- Supporter 跑位是常规传球前提
- 守门员动态防守独立收益高、实现简单
- 二过一风险最大、依赖最多，必须放最后

## 最后建议

15 天冲刺阶段，真正影响结果的不是“算法上限”，而是以下三件事：

- 角色切换是否稳定
- 失败后能否快速退回基础行为
- 场上联调时能否快速定位问题边界

因此建议始终坚持三条规则：

1. 每个协作功能都必须有超时退出。
2. 每个角色切换都必须有迟滞门限。
3. 每个高级战术失败后都必须回退到最基础的独立找球/追球。

这三条做到了，15 天内做出“能打”的版本是现实的。
